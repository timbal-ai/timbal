[2m──────────────────────────────────────────────────────────────────────────────────────────────────────────────
  Timbal: TestModel (sync handler, stateless). Built-in InMemory tracing always on.

  CA sync:  Crew.kickoff() in run_in_executor — what most teams run in production.
            kickoff_async() is NOT truly async: it is asyncio.to_thread(self.kickoff)
            (crewAIInc/crewAI crew.py). Same GIL + thread-safety limits apply.
            Burst capped at 5 threads: above that, instability from GIL contention
            and internal asyncio calls made from threads (GitHub #2632, #1234).

  CA async: Crew.akickoff() — native async path (added late 2024). Truly awaits
            all the way to litellm.acompletion(). Still mutates self.usage_metrics
            per run, so fresh Crew per concurrent call is required. Burst fully
            concurrent (no semaphore cap). Global event bus singleton and shared
            SQLite path remain potential bottlenecks at high concurrency (#2632).

  AgentOps: CrewaiInstrumentor().instrument() patches CrewAI methods globally and
            cannot be undone. Bare pass runs first, then AgentOps is activated.
            HTTP export is mocked — only hot-path span overhead is measured.

  Scenario 3: Timbal dispatches all 3 tools in one LLM call (parallel).
  CrewAI uses ReAct (one tool per LLM call) — 3 sequential steps instead.
